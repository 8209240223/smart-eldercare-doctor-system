## 一、本次改动目标

护士端定位为 **"护理执行和异常上报端"**，核心职责：护理记录、护理计划执行、体征采集、异常上报。

护士 **不应该** 做：处理/关闭预警、改预警规则、确认 AI 报告、发起正式转诊、审核护理记录、管理 AI 配置。

按手册完成：
1. 角色级接口权限（后端真权限）。
2. 菜单精简（前端移除医生决策类入口）。
3. 写按钮显隐（前端与后端注解对齐，避免出现"前端能点后端 403"）。

---

## 三、后端改动（按 Controller 列出）

### 3.1 NurseDashboardController（`/api/nurse/dashboard`）

**类级**：`@RequireRole({1,2,3})`

**依据**：护士工作台 = 护士使用、医生可查看摘要、管理员监管。端点全为 GET（`stats` / `tasks`），纯只读。**不能**写成 `{1,3}` 把医生排除。

**验收**：护士/医生/管理员登录均能调通 `GET /api/nurse/dashboard/stats`。

---

### 3.2 NursePlanController（`/api/nurse/plans`）

**类级**：`@RequireRole({1,2,3})`

**写方法（均收窄到护士 `{3}`）**：

| 端点 | 注解 | 说明 |
| --- | --- | --- |
| `POST /api/nurse/plans` | `@RequireRole({3})` | 新增护理计划 |
| `PUT /api/nurse/plans/{id}` | `@RequireRole({3})` | 修改护理计划 |
| `DELETE /api/nurse/plans/{id}` | `@RequireRole({3})` | 删除护理计划 |
| `PUT /api/nurse/plans/{id}/status` | `@RequireRole({3})` | 更新计划状态 |
| `POST /api/nurse/plans/{id}/increment` | `@RequireRole({3})` | 完成次数 +1 |

**读方法**（保持类级 `{1,2,3}`，管理员监管、医生审核都要看）：
- `GET /api/nurse/plans`（列表）
- `GET /api/nurse/plans/{id}`（详情）
- `GET /api/nurse/plans/stats`（统计）

**验收**：
- 护士能新增/修改/删除/开始/完成；医生/管理员调写接口返回 403，但可查列表/详情/统计。

### 3.4 VitalSignController（`/api/vitals`）

**类级**：`@RequireRole({1,2,3})`

**写方法分三档**：

| 端点 | 注解 | 说明 |
| --- | --- | --- |
| `POST /api/vitals/upload` | `@RequireRole({2,3})` | 护士为主、医生可录 |
| `POST /api/vitals/devices` | `@RequireRole({2,3})` | 绑定设备 |
| `PUT /api/vitals/devices/{id}/unbind` | `@RequireRole({2,3})` | 解绑设备 |
| `POST /api/vitals/mock/{elderId}` | `@RequireRole({1})` | **仅管理员**造数（测试） |
| `POST /api/vitals/mock/{elderId}/abnormal` | `@RequireRole({1})` | **仅管理员**造异常（测试） |

⚠️ **mock 造数端点风险**：能一键生成异常体征触发预警。生产环境**应下线**（可加开关或注释），目前仅靠角色注解限制。

**读方法**（保持类级 `{1,2,3}`）：
- `GET /api/vitals/devices/{elderId}`（设备列表）
- `GET /api/vitals/trend/{elderId}`（趋势）
- `GET /api/vitals/latest/{elderId}`（最新）

**验收**：
- 护士能上传/绑定/解绑；医生同上；管理员调上传/绑定/解绑 403。
- 医生/护士调 mock 造数 403；管理员可造数。

### 4.2 护士专属页写按钮显隐

合并规则：若按钮已有 `v-if`，用 `&&` 追加，不覆盖原条件。

#### 4.2.1 `vitals` 页（生命体征）

| 元素 | 改前 | 改后 | 依据 |
| --- | --- | --- | --- |
| 头部"新增绑定设备" | 无 v-if | `v-if="!isAdmin"` | 后端 `{2,3}` |
| 表内"解绑" | `v-if="Number(row.bindStatus)===1"` | `v-if="Number(row.bindStatus)===1 && !isAdmin"` | 后端 `{2,3}` |
| 筛选区"生成模拟数据" | 无 v-if | `v-if="isAdmin"` | 后端 `{1}`（mock 端点仅管理员） |

#### 4.2.2 `nurse-records` 页（护理记录）

护理记录写方法后端为 `{3}`（仅护士），前端写按钮统一 `v-if="isNurse"`。

| 元素 | 改前 | 改后 |
| --- | --- | --- |
| 头部"新增护理记录" | 无 v-if | `v-if="isNurse"` |
| 表内"编辑" | `v-if="row.reportStatus===0"` | `v-if="row.reportStatus===0 && isNurse"` |
| 表内"上报异常" | `v-if="row.isAbnormal===0\|\|row.reportStatus===0"` | `v-if="(row.isAbnormal===0\|\|row.reportStatus===0) && isNurse"` |
| 表内"删除" | `v-if="row.reportStatus===0"` | `v-if="row.reportStatus===0 && isNurse"` |
| 表内"详情" | 无 v-if | 保持不变（所有角色可见） |

#### 4.2.3 `nurse-plans` 页（护理计划）

| 元素 | 改前 | 改后 |
| --- | --- | --- |
| 头部"新增护理计划" | 无 v-if | `v-if="isNurse"` |
| 表内"编辑" | `v-if="row.status===0\|\|row.status===1"` | `v-if="(row.status===0\|\|row.status===1) && isNurse"` |
| 表内"开始执行" | `v-if="row.status===0"` | `v-if="row.status===0 && isNurse"` |
| 表内"完成一次" | `v-if="row.status===1"` | `v-if="row.status===1 && isNurse"` |
| 表内"删除" | `v-if="row.status===0\|\|row.status===1"` | `v-if="(row.status===0\|\|row.status===1) && isNurse"` |
| 表内"详情" | 无 v-if | 保持不变 |

## 八、风险与回滚

### 风险

| 风险 | 触发条件 | 影响 |
| --- | --- | --- |
| 所有接口 403 | `RoleInterceptor` 顺序错（早于 Jwt） | 全系统不可用 |
| 登录接口被拦 | 误给 AuthController 贴注解 | 登录失败 |
| 菜单白屏 | `NURSE_TAB_META` 删除时破坏数组语法 | 护士端导航错乱 |
| 护士点不到写按钮 | `v-if` 条件写反 | 仅 UI 异常，后端 403 一致 |

### 回滚

因改动集中在 7 个文件，回滚直接 `git revert` 本次提交即可。若只回滚护士端，保留地基改动，按本文件第五节"修改"列表逐一恢复：

1. 移除 4 个 Controller 的 `@RequireRole` 类级与方法级注解、`import`。
2. 恢复 `NURSE_TAB_META` 旧内容（把 `followup`/`assessment` 加回去）。
3. 恢复 app.js 中各 `v-if="...isNurse"` 与 `v-if="...isAdmin"` 为原状。
4. **不要**回滚地基（`RequireRole.java` / `RoleInterceptor.java` / `WebMvcConfig.java`），否则管理员/医生端未做也无影响。
