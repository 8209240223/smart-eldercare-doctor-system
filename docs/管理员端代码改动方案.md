## 一、目标与范围

### 1、改造目标

| 目标 | 说明 |
| --- | --- |
| 后端真实权限 | 在管理员归属的 Controller 上贴 `@RequireRole` 注解，挡住越权写请求 |
| 前端 UI 一致 | 隐藏管理员不应执行的"写操作"按钮，仅保留查看入口 |
| 代码质量提升 | 优化管理员端相关 Service/Controller 的代码结构、清理冗余、增强安全 |
| 性能与可扩展 | 在不破坏现有业务的前提下，提升响应速度与可维护性 |

### 2、改造范围（明确划界）

| 范围内（要做） | 范围外（不做） |
| --- | --- |
| 公共地基（第一块）相关改造 | 医生端业务页面（仅由第三块执行） |
| 管理员归属的 Controller 加注解 | 护士端业务页面（仅由第四块执行） |
| 管理员端菜单确认与 AI 配置入口 | 数据级权限（归属字段、过滤、token 注入等） |
| 管理员端 UI 隐藏写按钮 | 数据库 schema 变更（保持现状） |
| 管理员端代码结构优化、安全加固 | 转诊模块 `assertPermission` 内部逻辑（保持现状） |
| 改进优化方案文档 | AI 模型/算法改造 |

### 3、硬约束（绝对不能动）

1. 不删除任何已有业务功能（包括菜单项、API 端点）。
2. 不修改 `ReferralServiceImpl.assertPermission`（属于二期）。
3. 不动数据级权限相关 ServiceImpl 的 list 查询。
4. 不改数据库结构、不改 entity 字段。
5. 改动必须可独立回滚（每块分清边界）。

## 三、详细改动方案（按文件划分）

### 1、公共地基（第一块，依赖前置）

#### 1.1 新建注解 `RequireRole.java`

**文件**：`src/main/java/com/medical/common/annotation/RequireRole.java`（新建）

**内容**：

```java
package com.medical.common.annotation;

import java.lang.annotation.ElementType;
import java.lang.annotation.Retention;
import java.lang.annotation.RetentionPolicy;
import java.lang.annotation.Target;

/**
 * 角色权限注解。可贴在 Controller 类上（整类生效）或方法上（方法覆盖类）。
 * value 写允许访问的 userType：1=管理员 2=医生 3=护士。
 * 例：@RequireRole({2}) 仅医生；@RequireRole({1,2}) 管理员和医生。
 * 不贴 = 只要登录就能访问（保持旧行为）。
 */
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireRole {
    int[] value();
}
```

**实现逻辑**：标准 Java 运行时注解，`@Target` 同时支持类和方法。

**预期效果**：后续 Controller 只需一行注解即可限权。

**风险评估**：无，新建文件不破坏现有逻辑。

#### 1.2 新建拦截器 `RoleInterceptor.java`

**文件**：`src/main/java/com/medical/common/interceptor/RoleInterceptor.java`（新建）

**内容**：

```java
package com.medical.common.interceptor;

import com.fasterxml.jackson.databind.ObjectMapper;
import com.medical.common.annotation.RequireRole;
import com.medical.common.result.R;
import org.springframework.stereotype.Component;
import org.springframework.web.method.HandlerMethod;
import org.springframework.web.servlet.HandlerInterceptor;

import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import java.io.IOException;

@Component
public class RoleInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        if ("OPTIONS".equalsIgnoreCase(request.getMethod())) return true;
        if (!(handler instanceof HandlerMethod)) return true;
        HandlerMethod hm = (HandlerMethod) handler;

        RequireRole ann = hm.getMethodAnnotation(RequireRole.class);
        if (ann == null) ann = hm.getBeanType().getAnnotation(RequireRole.class);
        if (ann == null) return true;

        Object userTypeObj = request.getAttribute("currentUserType");
        if (userTypeObj == null) {
            writeError(response, 401, "未登录或Token无效");
            return false;
        }
        int userType = Integer.parseInt(userTypeObj.toString());
        for (int allowed : ann.value()) {
            if (allowed == userType) return true;
        }
        writeError(response, 403, "当前角色无权访问该功能");
        return false;
    }

    private void writeError(HttpServletResponse response, int code, String msg) throws IOException {
        response.setContentType("application/json;charset=utf-8");
        response.setStatus(200);
        response.getWriter().write(new ObjectMapper().writeValueAsString(R.fail(code, msg)));
    }
}
```

**实现逻辑**：

1. 读方法/类上的 `@RequireRole` 注解（方法优先于类）。
2. 从 `request` 取 `currentUserType`（由 `JwtInterceptor` 注入）。
3. 不在允许列表 → 返回 403；未登录 → 返回 401。

**预期效果**：所有受保护接口越权时返回 403。

**风险评估**：

- 顺序错误（角色拦截在 JWT 之前）会导致拿不到 userType，因此注册时必须 `order(2)`，JWT 用 `order(1)`。
- OPTIONS 放行：避免 CORS 预检被误拦。

#### 1.3 注册拦截器 `WebMvcConfig.java`

**文件**：`src/main/java/com/medical/common/config/WebMvcConfig.java`（修改）

**改动 1**：注入 `RoleInterceptor`。

```java
@Autowired
private JwtInterceptor jwtInterceptor;

@Autowired
private com.medical.common.interceptor.RoleInterceptor roleInterceptor;
```

**改动 2**：`addInterceptors` 方法替换为双拦截器注册。

```java
@Override
public void addInterceptors(InterceptorRegistry registry) {
    String[] excludes = {
            "/api/auth/login", "/api/auth/register", "/api/auth/captcha",
            "/api/auth/resetPassword", "/api/warnings/stream",
            "/", "/index.html", "/static/**", "/pages/**",
            "/css/**", "/js/**", "/img/**", "/lib/**"
    };
    registry.addInterceptor(jwtInterceptor)
            .addPathPatterns("/api/**").excludePathPatterns(excludes).order(1);
    registry.addInterceptor(roleInterceptor)
            .addPathPatterns("/api/**").excludePathPatterns(excludes).order(2);
}
```

**预期效果**：地基装好。`@RequireRole` 机制立即生效，但**尚未给任何 Controller 贴注解** → 行为零变化。

**风险评估**：中。若 excludes 漏项，登录会被拦；务必和原配置完全一致。

### 4、管理员端代码质量优化

#### 4.1 `AiConfigController` 异常处理优化

**改动**：

- 引入 `@Slf4j`（或 `LoggerFactory.getLogger(...)`）。
- 替换 `e.printStackTrace()` 为 `log.error("批量更新AI配置失败", e)`。
- `R.fail("配置值不能为空")` 改为 `R.fail(400, "配置值不能为空")` 统一响应码。

**预期效果**：

- 日志可被日志框架收集。
- 前端可按 code 做统一错误处理。

**风险评估**：低。`R.fail` 的两个重载均存在（已有代码使用过 400 状态码）。

---

#### 4.2 `ProfileController` 增强

**改动**：

1. `getAttribute("currentUserId")` 为 null 时增加 401 防御（除现有 `updateProfile` 外）。
2. `unreadCount(userId)` 等"查他人"接口增加 `currentUserId` 强制覆盖，防止越权查别人消息。
3. `messages` 等支持 `userId` 参数的接口，默认按登录用户过滤；管理员查全局时使用专用管理接口（不在本轮范围）。

**预期效果**：避免当前端传错 `userId` 时数据泄露。

**风险评估**：中。可能影响管理员代查功能；需保留"管理员可查所有"路径（建议加 `if (isAdmin) 不强制覆盖`）。

---

#### 4.3 `TimelineController` 入参校验

**改动**：

- `startDate` / `endDate` 格式校验（`yyyy-MM-dd`）。
- `pageNum` / `pageSize` 范围限制（1-100）。
- 入参非法时返回 400 而非空结果。

**预期效果**：减少异常 SQL 与不必要查询。

**风险评估**：低。

---

#### 4.4 `DashboardEnhancedController` 性能优化

**改动**：

1. `/api/dashboard/todo` 当前固定传 `2L`（硬编码医生 ID），改为从 `request.getAttribute("currentUserId")` 取登录用户，**仅医生传**；管理员/护士调此接口返回 403 或空数据（结合 `@RequireRole({1,2,3})` 但 `/todo` 是医生专属）。
2. `/api/dashboard/review-counts` 增加 Redis 缓存（5 秒），减少数据库压力。
3. `/api/dashboard/chronic-overview` 入参支持 `community` 过滤。

**预期效果**：

- 修复 todo 接口对所有角色返回相同硬编码数据的问题。
- 高频轮询场景数据库压力下降。

**风险评估**：中。todo 接口调整可能影响护士工作台（其内部也调 `/dashboard/todo`），需在护士端也确认是否需要工作台待办。建议保留 `2L` 默认值但增加"非医生时取自身"逻辑。

**风险评估**：中。可能影响"管理员帮用户操作消息"流程；建议仅拦截非管理员的跨用户操作。

#### 5.4 操作日志增强

**风险评估**：低。

### 6、性能优化

| 模块 | 优化点 | 实现方式 | 预期收益 |
| --- | --- | --- | --- |
| Dashboard | 统计接口缓存 | Redis 缓存 5-10s | 数据库 QPS 降 30%-50% |
| Timeline | 大数据量分页优化 | 强制 `pageSize ≤ 100` | 避免单查询拉满表 |
| AI Config | 减少重复 `reload()` | 批量更新合并为一次 reload | 减少 Redis 写入 |
| Profile | 头像上传异步化 | 同步改为异步落盘 | 主流程不被 IO 阻塞 |
| RiskProfile | 全量计算异步化 | 改为 `@Async` + 进度查询 | 接口响应 < 1s |

> 注：以上性能优化在 P3 范围，本轮**只做 Dashboard 缓存 + Timeline 分页上限**两项最小必做。

## 六、风险评估与应对

| 风险 | 等级 | 应对 |
| --- | --- | --- |
| 拦截器顺序错导致全 403 | 高 | 注册时严格 order(1)/order(2)；启动后立即 curl 验证 |
| 菜单删除误操作导致白屏 | 中 | 本轮不删菜单，仅确认 |
| v-if 条件写反导致按钮该显不显 | 中 | 充分测试三角色登录；与手册保持一致 |
| 改 `assertPermission` 破坏医生所有权 | 高 | 严格不动 `ReferralServiceImpl.assertPermission` |
| 性能优化引入并发问题 | 中 | 仅做最小集；异步操作加锁 |
| 数据库连接耗尽 | 低 | 增加分页上限 |
| 测试数据无护士账号 | 低 | 通过 SQL 注入或注册接口新建 |
