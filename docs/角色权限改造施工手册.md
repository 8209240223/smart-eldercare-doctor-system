### 给模型的话术模板

### 模型必须遵守的硬规则

1. **一次只做一张任务卡**，做完验收再做下一张。
2. **只改任务卡写明的文件和位置**，不顺手改别处、别的格式、别的注释。
3. **不删除已有业务功能**，本手册只加“权限校验”和“菜单/按钮显隐”。
4. 找位置用**搜索“定位锚点”字符串**，不要靠行号。
5. 每张卡改完要能**编译通过**再继续。
6. 若某处已被别的块改过（比如已有 `v-if`），**合并条件**而不是覆盖（见第五块冲突规则）。

### 权限的两层含义（每块都会涉及）

- **后端接口层**：贴 `@RequireRole` 注解，别人绕过前端直连接口也会被 403 拦住。**这是真权限。**
- **前端显示层**：按角色显隐菜单和按钮，让界面干净。**只做前端 = 假权限。**

### ⚠️ 本手册只做“角色级权限”，不做“数据级权限”（务必看懂再动手）

这是两件**完全不同**的事，别搞混：

| | 角色级权限（本手册要做） | 数据级权限（本手册**不做**，二期） |
| --- | --- | --- |
| 管的是 | 这个**角色**能不能访问这个功能 | 同一角色内，每个人只能看/改**自己名下**的数据 |
| 例子 | 护士不能调医生的预警处理接口 | 医生 A 只能看自己负责的老人、护士 A 只看自己录的记录 |
| 怎么做 | `@RequireRole` 注解 + 前端显隐 | 改各 ServiceImpl 的 list 查询、加责任关系表 |
| 本轮 | ✅ 做 | ❌ 不做 |

**当前系统的真实状态是「角色分离 + 数据共享池」**：

- 老人档案：所有医生看全库老人（`ElderServiceImpl` 的 doctorId 是前端可选参数，不强制过滤）。
- 预警：全局广播共享池，任何医生看/处理任何预警。
- 随访、护理记录：列表不按登录用户过滤，同角色互相可见。
- 老人与护士之间没有归属关系（`elder_info` 无 `nurse_id`，也无关系表）。

**给执行者的硬性要求**：

1. **不要去改任何 ServiceImpl 的 list/查询方法**，不要试图“让医生只看自己的老人”。那是二期的事，本轮做了反而会破坏现有功能。
2. 本手册所有 `@RequireRole` 只解决“**角色**能不能进这个接口”，不解决“**谁的数据**”。贴完注解后，同角色之间数据仍然共享，这是**预期行为，不是 bug**。
3. 详细的二期规划（责任关系表、归属字段从 token 注入、list 强制过滤等）见设计文档 [角色职责与功能边界划分.md](角色职责与功能边界划分.md) 第八节.3，本轮**不实现**。

# 第一块 · 公共地基（必须最先做，只做一次）

## 这一块要解决什么

现在系统所有接口只校验“是否登录”（`JwtInterceptor`），不区分角色。只有 `ReferralServiceImpl` 里手写了角色判断。
本块建立一套**统一的角色校验机制**：一个注解 `@RequireRole` + 一个拦截器。做完后，后面每个 Controller 只要加一行注解就能限权。

**已确认的事实（供理解，不用改）**：`JwtInterceptor` 已在每个请求里执行 `request.setAttribute("currentUserType", userType)`，所以拦截器能直接拿到当前角色。

## 任务卡 地基-1：新建注解 `@RequireRole`

**新建文件**：`src/main/java/com/medical/common/annotation/RequireRole.java`
**完整内容**（整段照抄）：

```java
package com.medical.common.annotation;

import java.lang.annotation.ElementType;
import java.lang.annotation.Retention;
import java.lang.annotation.RetentionPolicy;
import java.lang.annotation.Target;

/**
 * 角色权限注解。可贴在 Controller 类上（整类生效）或方法上（方法覆盖类）。
 * value 写允许访问的 userType：1=管理员 2=医生 3=护士。
 * 例：@RequireRole({2}) 仅医生；@RequireRole({1,2}) 管理员和医生。
 * 不贴 = 只要登录就能访问（保持旧行为）。
 */
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireRole {
    int[] value();
}
```
**验收**：文件存在，编译通过。

## 任务卡 地基-2：新建拦截器 `RoleInterceptor`

**新建文件**：`src/main/java/com/medical/common/interceptor/RoleInterceptor.java`
**完整内容**（整段照抄）：

```java
package com.medical.common.interceptor;

import com.fasterxml.jackson.databind.ObjectMapper;
import com.medical.common.annotation.RequireRole;
import com.medical.common.result.R;
import org.springframework.stereotype.Component;
import org.springframework.web.method.HandlerMethod;
import org.springframework.web.servlet.HandlerInterceptor;

import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import java.io.IOException;

/**
 * 角色权限拦截器。必须注册在 JwtInterceptor 之后。
 */
@Component
public class RoleInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        if ("OPTIONS".equalsIgnoreCase(request.getMethod())) {
            return true;
        }
        if (!(handler instanceof HandlerMethod)) {
            return true;
        }
        HandlerMethod hm = (HandlerMethod) handler;

        RequireRole ann = hm.getMethodAnnotation(RequireRole.class);
        if (ann == null) {
            ann = hm.getBeanType().getAnnotation(RequireRole.class);
        }
        if (ann == null) {
            return true; // 没贴注解 = 不限角色，保持旧行为
        }

        Object userTypeObj = request.getAttribute("currentUserType");
        if (userTypeObj == null) {
            writeError(response, 401, "未登录或Token无效");
            return false;
        }
        int userType = Integer.parseInt(userTypeObj.toString());
        for (int allowed : ann.value()) {
            if (allowed == userType) {
                return true;
            }
        }
        writeError(response, 403, "当前角色无权访问该功能");
        return false;
    }

    private void writeError(HttpServletResponse response, int code, String msg) throws IOException {
        response.setContentType("application/json;charset=utf-8");
        response.setStatus(200);
        ObjectMapper mapper = new ObjectMapper();
        response.getWriter().write(mapper.writeValueAsString(R.fail(code, msg)));
    }
}
```
**验收**：文件存在，编译通过。

## 任务卡 地基-3：注册拦截器到 `WebMvcConfig`

**修改文件**：`src/main/java/com/medical/common/config/WebMvcConfig.java`

**改动 1** —— 定位锚点搜 `private JwtInterceptor jwtInterceptor;`，在其下方补一个注入：
```java
    @Autowired
    private JwtInterceptor jwtInterceptor;

    @Autowired
    private com.medical.common.interceptor.RoleInterceptor roleInterceptor;
```

**改动 2** —— 定位锚点搜 `public void addInterceptors(InterceptorRegistry registry) {`，把整个方法替换为：
```java
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        String[] excludes = {
                "/api/auth/login",
                "/api/auth/register",
                "/api/auth/captcha",
                "/api/auth/resetPassword",
                "/api/warnings/stream",
                "/",
                "/index.html",
                "/static/**",
                "/pages/**",
                "/css/**",
                "/js/**",
                "/img/**",
                "/lib/**"
        };
        registry.addInterceptor(jwtInterceptor)
                .addPathPatterns("/api/**").excludePathPatterns(excludes).order(1);
        registry.addInterceptor(roleInterceptor)
                .addPathPatterns("/api/**").excludePathPatterns(excludes).order(2);
    }
```
> 注意：`excludes` 名单必须和原来的一致（登录、验证码、SSE 流、静态资源）。别漏项。

**验收**：编译通过；启动成功；此时**还没给任何 Controller 贴注解**，用任意账号登录，所有页面行为应和改造前**完全一样**（零变化）。这一步只是把机制装上、还没启用。

## 任务卡 地基-4：确认前端角色变量存在（只读检查，不改代码）

打开 `src/main/resources/static/pages/app.js`，搜 `isNurse` 和 `isAdmin`，确认 `data()` 里已有这两个变量（应该形如 `isNurse: roleType === 3` / `isAdmin: roleType === 1`）。

# 第二块 · 管理员端

> 前置：第一块公共地基必须已完成。

## A. 设计说明（管理员端定位）

管理员端 = **系统级管理和监管端**，不承担具体医疗业务执行。

**管理员应该负责**：
1. 全局数据看板（用户/老人/预警/随访/体检/评估总量）。
2. 用户与角色监管（医生、护士、管理员数量与状态）。
3. AI 配置管理（DeepSeek/GLM/Mock 开关、模型、超时、每日调用次数）。
4. 全局老人档案**查看**（默认不直接改业务内容）。
5. 全局预警**监管**（分布、处理状态、实时情况）。
6. 全局业务统计（随访、干预、评估、体检、转诊、护理审核）。
7. 系统配置与审计（操作日志、登录日志、消息通知）。

**管理员不应该做**（这些写操作后端会拦，前端要隐藏按钮）：
1. 直接处理健康预警。
2. 直接创建随访/干预记录。
3. 直接审核护士护理记录。
4. 直接创建护理计划/记录。
5. 直接生成/确认医疗评估结论。

一句话：管理员**能看、能监管、能配置 AI，不当业务经办人**。

# 第三块 · 医生端

> 前置：第一块公共地基必须已完成。
> 本块最重：医生是系统核心，且**共享业务页面的按钮显隐统一在本块处理**（因为这些页面主要是医生操作）。

## A. 设计说明（医生端定位）

医生端 = **医疗决策和闭环处理端**，核心是诊疗判断、风险识别、随访干预、护士审核。

**医生应该负责**：
1. 医生工作台。
2. 老人档案管理（查看+维护基础/健康档案、病史、用药、过敏、家族史）。
3. 重点人群管理（风险分层查看）。
4. 健康预警处理（查看详情、处理中/完成/忽略）。
5. 预警规则管理。
6. 随访计划管理（创建、按风险生成、维护状态）。
7. 随访记录管理。
8. 干预管理（宣教/用药指导/康复/心理）。
9. 评估记录（健康/慢病/心理/营养/康复/综合）。
10. AI 健康评估（生成、深度分析、确认、驳回）。
11. 体检管理（登记、查看、对比）。
12. 转诊协同（发起、接收、完成、取消/驳回）。
13. 护士审核（审核护士上报的异常护理记录和护理计划）。
14. 服务时间线查看。

**医生不应该做**：护士日常护理记录录入、护理计划执行进度维护、平台级 AI 配置、系统级用户管理。

- **HealthDetailController**：所有写方法（病史/用药/过敏/家族史的 add/update/delete）→ `@RequireRole({2})`；`@GetMapping("/{elderId}")` 不动。
- **ExamController**：`@PostMapping`(录入)、`@PutMapping("/{id}")`(修改) → `@RequireRole({2,3})`（医生+护士可录，管理员只读）；`@DeleteMapping("/{id}")` → `@RequireRole({2})`（删除属医生权限，护士不删）；Get 不动。
- **WarningRuleController**（类级 `{1,2}`）：写方法必须收窄到医生 —— `@PostMapping`(create)、`@PutMapping("/{id}")`(update)、`@DeleteMapping("/{id}")`、`@PutMapping("/{id}/toggle")`、`@PostMapping("/evaluate")` → 各加 `@RequireRole({2})`。依据：矩阵“预警规则=医生维护、管理员仅监管(只读)”，不能让管理员改规则。`@GetMapping` 保持类级 `{1,2}`。
- **ReviewController**（类级 `{1,2}`）：审核写动作**必须**加 `@RequireRole({2})`（**不是可选**）—— `@PostMapping("/records/{id}/approve")`、`@PostMapping("/records/{id}/reject")`、`@PostMapping("/plans/{id}/approve")`、`@PostMapping("/plans/{id}/reject")`。依据：矩阵“护士审核=医生审核、管理员仅监管”，管理员不可审核。`@GetMapping`(records/plans/stats) 保持类级 `{1,2}` 供管理员监管查看。
- **FollowupTaskController**：类级已是 `{1,2}`，护士已被挡住。写方法（generate/finish/cancel）默认医生用，无管理员写需求；如需严格可给写方法加 `@RequireRole({2})`。
- **ReferralController**：类级 `{1,2}`。写方法（accept/complete/reject/cancel/create）按矩阵“转诊：管理员仅监管、不处理”应加 `@RequireRole({2})` 收窄到医生。⚠️ 注意：`ReferralServiceImpl.assertPermission` 里目前对管理员是**全放行**的（`userType==1` 直接 return），与矩阵冲突。**本轮处理方式：只在 Controller 写方法加 `@RequireRole({2})` 注解，把管理员挡在写方法之外**（请求进不到 Service，那段全放行逻辑走不到，无害）。**不要去改 `assertPermission` 方法本身**——修正它属于二期数据权限专项，本轮保持不动，以免影响医生的所有权校验（发起方/接收方医生）。

**后端验收**：编译通过；护士账号直连 `PUT /api/warnings/1/handle`、`PUT /api/warnings/1/processing` 均返回 403；管理员直连 `POST /api/warning-rules`、`POST /api/review/records/1/approve`、`POST /api/referrals` 均返回 403；医生账号可正常处理预警、改规则、审核、发起转诊。

# 第四块 · 护士端

> 前置：第一块公共地基必须已完成。

## A. 设计说明（护士端定位）

护士端 = **护理执行和异常上报端**，核心是护理记录、护理计划执行、体征采集、异常上报。

**护士应该负责**：
1. 护士工作台（今日护理记录、进行中计划、待处理异常、老人总数）。
2. 护理记录管理（新增/修改/删除，记录内容、观察、评价）。
3. 异常护理上报（上报给医生审核）。
4. 护理计划管理（新增/修改/删除、开始执行、完成一次）。
5. 护理计划执行进度维护。
6. 老人档案**查看**（不改核心医疗诊断字段）。
7. 生命体征采集（血压/血糖/心率/血氧/体温）。
8. 预警中心**查看**（不做最终处理）。
9. 服务时间线查看。
10. AI 评估**辅助查看**（不确认/驳回医疗结论）。

**护士不应该做**：处理/关闭预警、改预警规则、确认 AI 报告、发起正式转诊、审核护理记录、管理 AI 配置。

一句话：护士**执行、记录、采集、上报**；决策留给医生。

## 冲突规则（多块改了同一文件时）

只有 `app.js` 和 `WebMvcConfig.java` 会被多块碰到。规则：

1. **WebMvcConfig.java**：只有第一块（地基-3）改它，其他块不碰 → 无冲突。
2. **app.js 的菜单常量**：三组常量互相独立（`ADMIN_TAB_META` / `TAB_META` / `NURSE_TAB_META`），各块只改自己那组 → 无冲突。
3. **app.js 的共享页面按钮**（预警/随访/干预等）：**统一由第三块（医生端）处理**。护士块遇到同一处（如预警只读）**不重复改**，只验收。这是本手册特意的安排，避免两块改同一行。
4. 万一出现同一行既有 A 条件又要加 B 条件：用 `&&` 合并两个条件，**不要删掉已有条件**。

> **管理员（userType=1）除「AI 配置」和「个人中心」外，对所有业务模块一律只读。** 遇到任何写接口（`POST`/`PUT`/`DELETE`：新增/修改/删除/处理/审核/发起等），管理员都不在允许名单；管理员只能 `GET`。判断某个写方法要不要给管理员时，答案永远是“不给”。

### 代码核实的关键点（易错，务必记住）

1. **WarningController 有 5 个写端点**：handle / ignore / **processing** / read / create（`processing` 最易漏），全 `{2}`。
2. **AiHealthReportController 有 4 个写端点**：generate / **deep-analysis** / confirm / reject（`deep-analysis` 最易漏），全 `{2}`。
3. **RiskProfileController（重点人群）有 2 个 POST 写端点**：calculate 全量 / 单人，写 `{2}`、读 `{1,2,3}`。
4. **ReviewController 审核写方法必须 `{2}`**（不是可选）：approve/reject records、approve/reject plans。管理员不可审核。
5. **WarningRuleController 写方法必须 `{2}`**：create/update/delete/toggle/evaluate。管理员不可改规则。
6. **ReferralController 写方法应 `{2}`**：accept/complete/reject/cancel/create；注意 `assertPermission` 对管理员是全放行的，靠注解在前面挡住。
7. **VitalSignController mock 造数端点锁 `{1}` 且生产下线**：`/mock/{elderId}`、`/mock/{elderId}/abnormal`。
8. **NurseDashboardController 是纯只读**，类级 `{1,2,3}`（含医生），不要写成 `{1,3}`。
