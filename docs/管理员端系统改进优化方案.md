## 一、综述

本轮（一期）改造以"角色级权限"为主线，对管理员端系统做了 5 个 Controller 的注解贴标、前端写按钮显隐、代码质量优化、安全加固与最小集性能优化，并补齐了 30 个单元测试。当前所有改动通过编译、测试、启动验证。

**本期改造的硬性边界**：

1. 只做"角色级权限"（谁能进哪个接口），不做"数据级权限"（同角色内谁能看到哪些数据）。
2. 不删除任何业务功能、不改动数据库 schema、不动 `ReferralServiceImpl.assertPermission`。
3. 改动可独立回滚。

**本文档进一步梳理**：

- 一期已完成的项（✅ 标注，附位置）
- 一期可继续打磨的项（🔧 标注，本文档给出具体方案但暂未实现）
- 二期数据级权限的改进项（📅 标注，仅规划）
- 可扩展性建议（💡 标注）

### 2、可继续打磨的功能改进

#### 2.1 🔧 AI 配置：版本回滚能力

**现状**：`AiConfigService.set(key, value)` 直接写 Redis，无历史。

**改进建议**：

- 增加 `ai_config_history` 表（key, value, desc, operator_id, update_time）。
- 每次 `set` 前先 snapshot 到 history。
- 增加"查看历史"接口（限管理员）。
- 一键回滚到指定时间点的配置。

**实现路径**：

1. 新建表 `ai_config_history`。
2. `AiConfigServiceImpl.set` 改造（AOP 拦截即可）。
3. 新增 `AiConfigHistoryController`（`/api/ai/config/history`）。

**价值**：运维场景误操作可回退；审计追踪。

#### 2.2 🔧 个人中心：统一 my-profile 视图

**现状**：`ProfileController` 的 `updateProfile` 只改 `SysUser` 字段。

**改进建议**：

- 拆出独立的"我的资料"与"个人中心"两个端点。
- 我的资料：基本信息（仅自己可改）。
- 个人中心：含安全设置（密码/手机/邮箱）、消息、操作日志聚合。

**价值**：前端可按场景展示。

#### 2.3 🔧 消息中心：批量已读/未读 API 完善

**现状**：`/api/profile/messages` 返回分页列表。

**改进建议**：

- 筛选器：按 sourceType、msgType、isRead、时间范围。
- 按 type 批量标记已读。
- 删除单条/批量删除（限本人/管理员）。

**价值**：用户体验更完整。

#### 2.4 🔧 工作台：导出报表

**现状**：工作台数据仅展示。

**改进建议**：

- "导出 Excel" 按钮（限管理员）。
- 利用 easyexcel 库（已在 pom 中）实现。
- 文件直接下载到客户端，不落服务器。

**价值**：监管上报场景。

#### 2.5 🔧 时间轴：时间范围快捷选项

**现状**：`/api/timeline` 必须传 startDate/endDate。

**改进建议**：

- 前端增加"近 7 天 / 近 30 天 / 近 90 天 / 自定义"快捷选项。
- 后端无需改动（前端组装日期）。

**价值**：用户友好。

## 三、性能优化项

### 1、已完成的性能优化

| 优化 | 位置 | 效果 |
| --- | --- | --- |
| `@Cacheable` 加到 `getReviewCounts` | DashboardEnhancedServiceImpl | 缓存命中跳过 2 次 DB |
| `@Cacheable` 加到 `getChronicOverview` | DashboardEnhancedServiceImpl | 缓存命中跳过 6+ 次 DB |
| 时间轴 `pageSize` 上限 100 | TimelineController | 防止拉满表 |
| 时间轴 `elderId` 校验 | TimelineController | 减少无效查询 |

> **预期收益**：高频轮询场景下，dashboard 相关接口 QPS 减少 30-50%。

### 2、可继续优化的性能点

#### 3.2.1 🔧 AI 配置 Redis 序列化优化

**现状**：`AiConfigService` 直接存 Redis String，未用 Jackson 序列化。

**改进建议**：

- 用 `RedisTemplate` 统一封装 `ai:config:*` 键。
- 用 Jackson 序列化 Map 结构，可读性更好。
- `reload()` 改造为批量 mget。

**价值**：减少 Redis 网络往返。

#### 3.2.2 🔧 Dashboard 缓存预热

**现状**：首次访问 dashboard 接口触发 8+ 次 DB 查询。

**改进建议**：

- 应用启动时（或定时）主动加载 dashboard 数据到 Redis。
- 缓存键加 `version` 后缀，写操作时 bump version（自动失效）。
- 监控命中率（命中率 < 80% 触发告警）。

**价值**：首屏 < 100ms。

#### 3.2.3 📅 数据级权限下的 list 优化

**现状**：所有 list 走全表 + 内存过滤（极端情况）。

**改进建议（二期）**：

- 加 doctor_id / nurse_id 索引。
- ServiceImpl list 方法按 token 注入的 currentUserId 强制过滤（覆盖前端的可选参数）。
- 重点人群、老人档案、随访等模块首先改造。

**价值**：数据隔离 + 查询性能同步提升。

#### 3.2.4 🔧 预警 SSE 推送去重

**现状**：每条预警都通过 SSE 推给所有在线医生。

**改进建议**：

- 同一老人同一规则 5 分钟内的预警合并推送。
- SSE 消息体增加 `count` 字段。

**价值**：减少 SSE 通道占用。

#### 3.2.5 🔧 关键页面 SSR 化（可选）

**现状**：管理端工作台是 SPA，所有图表通过前端 API 渲染。

**改进建议**：

- 首屏用服务端渲染（SSR），图表静态化。
- 减少首屏 TTFB 50%。

**价值**：用户体验；非紧急。

#### 3.2.6 🔧 MyBatis-Plus 慢 SQL 监控

**现状**：未配置 SQL 执行时长监控。

**改进建议**：

- 集成 p6spy 或 Druid 自带 SQL 监控。
- 慢 SQL（>500ms）自动记录日志。
- 集成 Druid 监控面板（/druid，已在 pom 中）。

**价值**：持续性能调优。

## 四、安全加固措施

### 1、已完成的安全加固

| 加固 | 位置 | 效果 |
| --- | --- | --- |
| `@RequireRole` 类级/方法级 | 5 个 Controller | 越权写 403 |
| 角色权限矩阵在拦截器层强制 | `RoleInterceptor` | 即使绕过前端也拦住 |
| 消息归属校验 | `ProfileController.markRead` | 防止越权读他人消息 |
| 批量已读限管理员 | `ProfileController.markAllRead` | 防止误清空 |
| AI 配置 key/value 长度限制 | `AiConfigController` | 防止超长污染 |
| 头像文件扩展名校验 | `ProfileController.resolveAvatarExtension` | 防止伪扩展名 |
| `e.printStackTrace()` → log | `AiConfigController` | 防止 stacktrace 入侵响应 |
| 时间轴日期格式校验 | `TimelineController` | 防止 SQL 异常绕过 |
| 时间轴 pageSize 上限 | `TimelineController` | 防止大查询 |

### 2、可继续加固的安全点

#### 4.2.1 🔧 登录失败锁定时间提升

**现状**：账号被锁后只能等 TTL 自然过期。

**改进建议**：

- 连续 5 次失败后锁定 30 分钟。
- 累计 10 次失败需管理员解锁。
- 锁定期间返回 429 状态码（已部分实现）。

**实现路径**：

- `RedisKeyConstant.LOGIN_LOCK_TTL` 提升。
- `AuthServiceImpl.incrementLoginError` 加累计计数。

**价值**：抗暴力破解。

#### 4.2.2 🔧 JWT 黑名单（强制下线）

**现状**：JWT 一旦签发，到期前一直有效。

**改进建议**：

- 维护 `jwt:blacklist:<userId>` 集合（Redis Set）。
- 用户主动改密码/被踢出时，加入黑名单。
- `JwtInterceptor` 校验黑名单。

**价值**：支持即时吊销 token。

#### 4.2.3 🔧 头像上传增加水印/MIME 严格校验

**现状**：仅校验 ContentType。

**改进建议**：

- 上传后用 ImageIO 二次验证（实际读取图像头）。
- 大小写敏感：`.JPG` vs `.jpg` 统一。
- 加水印：用户 ID + 时间戳（防盗用）。

**价值**：防止图床被滥用。

#### 4.2.4 🔧 操作日志全量记录

**现状**：仅部分写接口有 `@OperationLog`（如 AI 配置、修改个人资料）。

**改进建议**：

- 所有写方法（POST/PUT/DELETE）一律加 `@OperationLog`。
- 记录字段：操作者、IP、URL、方法、入参摘要、结果。
- 增加"操作日志查询"管理端页面（仅管理员）。

**价值**：审计可追溯。

#### 4.2.5 🔧 XSS 防御

**现状**：用户输入字段（realName、remark、interventionContent）未做 HTML 转义。

**改进建议**：

- 全局增加 `JsonRequestBodyAdvice` 或 Filter。
- 入参 String 字段统一转义 `<>"'&`。
- 风险字段（description、remark、content）额外处理。

**价值**：防止 XSS 攻击。

#### 4.2.6 🔧 CSRF 防御

**现状**：未配置 CSRF token。

**改进建议**：

- 引入 `spring-security` 的 CSRF 保护（仅管理后台写接口）。
- 或自实现 CSRF token 机制（更轻量）。

**价值**：防止 CSRF 攻击。

#### 4.2.7 🔧 SQL 注入二次审查

**现状**：使用了 MyBatis-Plus 的 LambdaQueryWrapper，绝大部分场景安全。

**改进建议**：

- 静态扫描所有 `${}` 字面量替换（应使用 `#{}`）。
- 禁止在 `xml` mapper 中用 `${}`。
- 重点：搜索 `${` 关键字，零容忍。

**价值**：长期安全。

#### 4.2.8 📅 护士-老人归属关系（二期）

**现状**：护士对所有老人可见。

**改进建议（二期）**：

- `elder_info` 加 `nurse_id` 字段（迁移期可为空）。
- 新建 `elder_nurse_assign` 关系表（多对多）。
- 护士登录时 token 注入其分配的老人 ID 列表。
- 业务查询按列表强制过滤。

**价值**：数据级隔离。

## 五、可扩展性建议

### 1、💡 模块化拆分

**现状**：admin-dashboard 数据由 `DashboardEnhancedServiceImpl` 提供，混合了多种业务数据。

**改进建议**：

- 按业务域拆：`ElderDashboardService`、`WarningDashboardService`、`FollowupDashboardService`。
- 统一接口 `DashboardDataProvider`，由 Spring 注入 List，自动组装。
- 各 provider 独立演进。

**价值**：后续扩展不破坏现有。

### 2、💡 角色权限配置化

**现状**：`@RequireRole({1,2,3})` 硬编码。

**改进建议**：

- 数据库表 `role_permission` 存角色-接口对应关系。
- 启动时加载到内存（带缓存）。
- 自定义注解 `@PermissionCheck("admin:config:update")` 替代硬编码 userType。
- 权限变更不需改代码、不需重启。

**价值**：动态权限、零代码上线。

### 3、💡 前端组件化

**现状**：`app.js` 6000+ 行单体。

**改进建议**：

- 拆分为 `pages/elder.js`、`pages/warning.js`、`pages/admin-config.js` 等。
- 公共组件提取：`<StatCard>`, `<DataTable>`, `<ConfirmModal>`。
- 路由懒加载。

**价值**：维护性 + 首屏速度。

### 4、💡 统一审计

**新增 `SysAuditLog` 模块**：

- 所有"敏感操作"（审核、配置变更、转诊、AI 报告确认）记录到 audit log。
- 单独存储（可考虑 ClickHouse）。
- 管理员可查、可导出。

**价值**：合规、事后追溯。

### 5、💡 多租户预留

**现状**：所有数据共享一库。

**改进建议**：

- 所有表加 `tenant_id` 字段（迁移期可为空）。
- MyBatis-Plus 拦截器自动注入 tenant_id 过滤。
- 为 SaaS 化做准备。

**价值**：未来可上云服务多家养老机构。

### 6、💡 AI 配置外部化

**现状**：配置存 Redis，但模型切换、key 维护都在管理员页。

**改进建议**：

- 支持配置变更热更新（已实现 reload）。
- 增加"配置变更广播"：通过 Redis Pub/Sub 通知所有实例。
- 增加回滚操作（见功能改进 2.1）。

**价值**：多实例下零停机运维。

### 7、💡 监控指标暴露

**新增 `/actuator` 端点**：

- 集成 `spring-boot-starter-actuator`。
- 暴露 `/metrics`、`/health`、`/info`。
- 集成 Prometheus + Grafana（可选）。

**价值**：可观测性。
