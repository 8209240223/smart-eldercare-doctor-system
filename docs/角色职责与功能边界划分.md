# 智慧医养医生服务系统角色职责与功能边界划分

## 一、当前系统角色

当前项目主要存在三类登录角色：

1. 管理员：`userType = 1`
2. 医生：`userType = 2`
3. 护士：`userType = 3`

前端目前通过 `src/main/resources/static/pages/app.js` 中的三组菜单区分角色端：

1. 医生端：`TAB_META`
2. 管理员端：`ADMIN_TAB_META`
3. 护士端：`NURSE_TAB_META`

后端目前主要通过 JWT 拦截器判断“是否登录”，大多数业务接口没有统一做角色级权限控制。也就是说，现在系统已经有“前端分端菜单”，但后端职责隔离还不彻底。

## 一（补）、权限的两个层次（务必先读，避免误解本文档）

本文档讨论的“三端分离”涉及两个**完全不同**的权限层次，不能混为一谈：

### 层次 A：角色级权限（本轮改造的目标，可落地）

“**这个角色能不能访问这个功能/接口**”。例如：护士不能调用医生的预警处理接口、管理员不能改 AI 配置以外的业务。这一层靠后端统一的 `@RequireRole` 注解 + 前端菜单/按钮显隐实现，是本轮改造真正要做的事。

### 层次 B：数据级权限（当前**未实现**，列为二期规划）

“**同一角色内，每个人只能看/改属于自己的那部分数据**”。例如：医生 A 只能看自己负责的老人、只处理分配给自己的预警；护士 A 只能维护自己录入的护理记录。

**必须如实说明当前代码现状：除转诊模块外，系统没有做数据级隔离。** 具体表现为：

1. **老人档案是全局可见**：`elder_info` 虽有 `doctor_id`（责任医生）字段，但列表查询的 `doctorId` 是前端可选参数，不传就返回全库老人；医生 A 默认能看到所有老人。
2. **预警是全局共享池**：预警生成时不绑定医生，通过 SSE **广播给所有在线医生**，任何医生都能看到、处理任何预警。`doctor_id` 只是“谁处理了才回填谁”，不是归属。
3. **随访计划/记录列表根本没有 doctorId 过滤**：任何医生看到全部随访。
4. **护理记录不按护士过滤**：护士 A 能看到护士 B 的全部护理记录。
5. **老人与护士之间零关系**：`elder_info` 没有 `nurse_id`，也没有护士-老人关系表；系统当前无法知道“哪个护士负责哪些老人”。护士端的数据边界在模型层是未定义的。
6. **归属字段不可信**：护理记录/随访等在 create 时，`nurse_id`/`doctor_id` 来自前端请求体、而非从登录 token 注入，理论上可被前端伪造，暂不能作为隔离依据。

**因此本系统当前的真实模型是「角色分离 + 数据共享池」**：三端各自只操作本角色的功能，但同一角色内的数据是全员共享、无个人边界的。对于小规模医养系统，这个模型可以正常运行；数据级隔离作为**二期规划**，见第八节.3。**在二期到来之前，本文档第七节权限矩阵中的“可维护/可处理/可查看”均指“角色能否操作该功能”，不代表“只能操作自己名下的数据”。**

### 2、管理员应该负责的功能

管理员端应该定位为“系统级管理和监管端”，不应该承担具体医疗业务执行。

建议管理员保留以下职责：

1. 全局数据看板：用户总数、老人档案总数、预警数量、随访数量、体检数量、评估数量。
2. 用户与角色监管：查看医生、护士、管理员数量和基础状态。
3. AI 配置管理：DeepSeek、GLM、Mock 开关、模型、超时时间、每日调用次数等配置。
4. 全局老人档案查看：可以查看全平台老人信息，但默认不直接修改业务内容。
5. 全局预警监管：查看预警分布、处理状态、实时预警情况。
6. 全局业务统计：随访、干预、评估、体检、转诊、护理审核等业务统计。
7. 系统配置与审计：查看操作日志、登录日志、消息通知、系统异常。

### 3、管理员不建议负责的功能

以下功能不建议由管理员直接执行：

1. 直接处理健康预警。
2. 直接创建随访记录。
3. 直接写干预记录。
4. 直接审核护士护理记录。
5. 直接创建护理计划或护理记录。

管理员可以“查看和监管”，但不应该成为具体业务经办人。

### 2、医生应该负责的功能

医生端应该定位为“医疗决策和闭环处理端”，核心是诊疗判断、风险识别、随访干预和护士审核。

建议医生保留以下职责：

1. 医生工作台：查看本人相关老人、预警、随访、干预、审核事项。
2. 老人档案管理：查看和维护老人基础档案、健康档案、病史、用药、过敏、家族史。
3. 重点人群管理：查看高危、重点、关注、普通老人风险分层结果。
4. 健康预警处理：查看预警详情，标记处理中、处理完成、忽略预警。
5. 预警规则管理：维护血压、血糖、心率、血氧等规则，或至少由医生参与配置。
6. 随访计划管理：创建随访计划，按风险生成随访计划，维护随访计划状态。
7. 随访记录管理：记录随访结果、症状、指标、用药依从性、下次随访时间。
8. 干预管理：创建健康宣教、用药指导、康复训练、心理干预等记录。
9. 评估记录：健康评估、慢病评估、心理评估、营养评估、康复评估、综合评估。
10. AI 健康评估：生成 AI 健康评估报告，进行深度分析、确认、驳回。
11. 体检管理：登记和查看体检数据，做体检对比。
12. 转诊协同：发起转诊、接收转诊、完成转诊、取消或驳回转诊。
13. 护士审核：审核护士上报的异常护理记录和护理计划。
14. 服务时间线：查看老人全流程服务轨迹。

### 3、医生不建议负责的功能

以下功能不建议由医生直接执行：

1. 护士日常护理记录录入。
2. 护士护理计划的执行进度维护。
3. 护士工作台任务处理。
4. 平台级 AI 配置管理。
5. 管理员级用户统计和系统配置。

医生应该主要负责判断、审核、决策和医疗闭环。

## 五、当前职责分离不彻底的问题

### 1、前端菜单已经分角色，但后端权限没有完全同步

目前前端根据 `userType` 显示不同菜单，但后端多数接口只校验 JWT 登录状态，并没有强制限制角色访问。

风险是：

1. 医生可以通过接口直接调用护士接口。
2. 护士可以通过接口访问医生业务接口。
3. 管理员可以直接执行具体医疗业务。
4. 前端隐藏菜单不能真正防止越权访问。

### 2、护士端包含部分医生业务

护士端现在包含：

1. 预警中心
2. 随访计划
3. AI 评估

这些功能本身更偏医生决策。如果护士端要保留，建议改成“只读”或“辅助查看”，不能让护士执行最终医疗处理。

### 3、管理员端包含过多业务执行入口

管理员端现在包含老人档案、预警、重点人群、随访、干预、评估、转诊、体检等大量业务页面。

建议管理员端可以保留查看权限，但具体新增、修改、删除、处理、审核操作应该限制。

### 4、医生端和护士端的护理相关边界容易混

医生端有“护士审核”，护士端有“护理记录”和“护理计划”。这个方向是对的，但需要明确：

1. 护士负责提交。
2. 医生负责审核。
3. 管理员负责监管。

不要让护士审核自己的内容，也不要让医生直接替护士维护日常护理执行数据。

### 5、转诊模块已有角色校验，但其他模块没有统一实现

当前 `ReferralServiceImpl` 中已经出现医生、护士、管理员的角色权限判断，这是比较好的方向。

建议后续把这种权限校验抽成统一机制，用到所有核心业务模块。

### 管理员只读总则（判断权限时的第一原则）

通读全部 23 个 Controller 的真实端点后，可以提炼出一条最简判断规则：

> **管理员（userType=1）除了「AI 配置」和「个人中心」，对所有业务模块一律只读。系统中没有任何一个业务写操作应该开放给管理员。**

也就是说，遇到任何写接口（新增/修改/删除/处理/审核/发起等 `POST`/`PUT`/`DELETE`），管理员都不在允许名单里；管理员只能 `GET`（查看、统计、监管）。这条总则可以直接套用，避免逐个模块纠结。

### 基于代码核实的补充说明（矩阵之外、代码里才看得到的点）

以下几点是逐行阅读 Controller 后发现的、纸面矩阵无法体现的真实情况，作为落地判断依据：

1. **重点人群 RiskProfileController 有写端点。** 该模块除只读列表/统计外，还有两个写接口：`POST /api/risk/elders/calculate`（全量计算风险）、`POST /api/risk/elders/{elderId}/calculate`（单人计算）。按矩阵“重点人群：医生可处理”，这两个写端点应限医生，读端点三角色可看。

2. **护士审核 ReviewController 必须限医生写。** 审核动作 `approveRecord/rejectRecord/approvePlan/rejectPlan` 属于医生专有决策，管理员**不可**执行（否则违反“护士审核：管理员仅监管”）。类级即使放开给管理员读，写方法也必须收窄到医生。

3. **转诊 ReferralController 与代码现状冲突（需注意）。** 现有 `ReferralServiceImpl.assertPermission` 里对管理员（userType=1）是**全放行**的，这意味着代码层面管理员目前能发起/接收/完成转诊，与矩阵“转诊：管理员仅监管、不处理”冲突。按设计，转诊的写方法应限医生；后续改代码时可在写接口加角色注解拦住管理员（Service 内那段放行逻辑届时走不到，保留不影响），或一并调整 `assertPermission`。

4. **生命体征 VitalSignController 含测试造数端点。** 除正常录入/设备绑定外，还有 `POST /api/vitals/mock/{elderId}`、`POST /api/vitals/mock/{elderId}/abnormal` 两个 mock 端点，能一键生成异常体征触发预警。这类端点风险较高，建议**仅管理员**可用，并在生产环境下线；正常体征录入归护士（医生可录）。

5. **护士工作台 NurseDashboardController 是纯只读。** 其端点全部为 `GET`（stats/tasks），按矩阵“护士工作台：医生可查看摘要”，应允许医生查看，不应把医生排除在外。

## 八、后续代码拆分建议

### 1、前端拆分建议

当前前端主要集中在一个 `app.js` 中，角色页面、业务逻辑、弹窗、接口调用全部混在一起。

建议拆成：

1. `doctor-dashboard.js`：医生工作台。
2. `doctor-warning.js`：医生预警处理。
3. `doctor-followup.js`：医生随访闭环。
4. `doctor-review.js`：医生审核。
5. `nurse-dashboard.js`：护士工作台。
6. `nurse-record.js`：护理记录。
7. `nurse-plan.js`：护理计划。
8. `admin-dashboard.js`：管理员工作台。

11. `shared-profile.js`：个人中心。

### 2、后端权限拆分建议

建议新增统一角色权限控制：

1. `@RequireRole("ADMIN")`
2. `@RequireRole("DOCTOR")`
3. `@RequireRole("NURSE")`
4. `@RequireAnyRole({"ADMIN","DOCTOR"})`

然后在 Controller 层明确标注：

1. `/api/ai/config/**`：仅管理员。
2. `/api/review/**`：医生审核，管理员只读监管。
3. `/api/nurse/**`：护士维护，医生审核，管理员监管。
4. `/api/warnings/**`：医生处理，护士只读，管理员监管。
5. `/api/referrals/**`：医生处理，管理员监管，护士禁止处理。
6. `/api/followup/**`：医生维护，护士只读。

### 3、数据权限拆分建议（二期规划，本轮不做）

> ⚠️ 本小节描述的是**尚未实现**的目标状态，属于**二期规划**。当前系统（一期，本轮改造）只做角色级权限，数据级隔离一律不做。参见“一（补）、权限的两个层次”。

目标状态（二期要达到的，现在都还没有）：

1. 医生只能处理自己负责的老人、预警、随访和转诊。
2. 护士只能维护自己负责或被分配的护理记录、护理计划。
3. 管理员可以查看全局数据，但默认不参与业务经办。

要实现上述目标，二期需要依次补齐以下前置条件（缺一不可，这也是为什么本轮不做）：

1. **补护士-老人归属关系**：`elder_info` 增加 `nurse_id`，或新建 `elder_nurse` / `elder_doctor` 责任关系中间表。当前老人与护士之间零关联，护士的数据边界无从谈起。
2. **归属字段改由 token 注入**（对应 B1）：护理记录、护理计划、随访、评估等在 `create` 时，`nurse_id`/`doctor_id` 必须从登录用户的 token 取，而不是接收前端请求体里的值。否则归属字段可被伪造，做隔离也没意义。
3. **列表查询强制按当前用户过滤**：各 `ServiceImpl` 的 list 方法把“前端可选参数 doctorId/nurseId”改为“从 token 强制注入的 currentUserId”，医生/护士只能查到自己名下的数据。这一步会改动几乎所有业务 Service 的查询，风险最大，务必二期专项处理。
4. **预警/审核的归属策略需先定调**：预警当前是全局广播共享池、审核当前是全局待审核池。二期要明确它们到底是“共享池（谁都能处理）”还是“归属制（只处理分配给自己的）”，再决定是否改造。
5. **异常护理记录“两步上报”问题**（对应 B2，属数据流缺陷，可提前到本轮外单独修）：护士 `create` 一条 `isAbnormal=1` 的记录时，`reportStatus` 仍默认为 0，不会自动进入医生待审核池，必须护士再单独调 `/records/{id}/report` 才进池。二期或专项修复时应让异常记录创建即进池，避免漏报。

## 九、结论

当前系统已经具备医生、护士、管理员三端的雏形，但职责分离主要停留在前端菜单层。要让系统更像真实医养业务系统，分两期推进：

**一期（本轮改造）**：补齐**角色级**权限——后端统一 `@RequireRole` 注解 + 前端菜单/按钮显隐，让三端各自只能访问本角色的功能。这一期落地后，系统是「角色分离 + 数据共享池」模型。

**二期（后续规划）**：补齐**数据级**权限（见第八节.3），让同一角色内每个人只能操作自己名下的数据。前置条件较多、改动面大，不与一期混做。

最终定位应该是：

1. 管理员：系统监管和配置。
2. 医生：医疗判断、预警处理、随访干预、评估审核。
3. 护士：护理执行、记录维护、体征采集、异常上报。
